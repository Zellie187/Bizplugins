# Performance Tests

**Status: not implemented.** No load test, benchmark, or performance regression suite exists anywhere in this repository. No performance characteristics of this plugin (REST endpoint latency under load, `WorkflowRepository` query performance at scale, `dbDelta()` migration time on a large `bizhub_workflow_transitions` table) have been measured. As with `Acceptance-Tests.md`, this gap exists because no running WordPress/MySQL environment was available in the environment this codebase was developed and verified in — PHPUnit's 32 passing tests all run in milliseconds against in-memory fakes, which says nothing about performance against a real, populated database.

## Where performance risk plausibly exists, based on the actual schema and code

- **`WorkflowRepositoryInterface::history()`** is called every time `WorkflowRepository::find()` hydrates an instance (`find()` always calls `withHistory($this->history($uuid))`). For a workflow instance with an unusually long transition history, this means every `GET /company-registrations/{uuid}` call (and every `transition()`/`rollback()`, which also calls `find()` internally via `findOrFail()`) pays the cost of loading the *entire* history, indexed only by `workflow_uuid` (see `docs/database/Indexes.md`) — there is no pagination or "just give me the current state, not history" fast path at the repository level (the REST layer's `show()` method does control whether history is *returned* to the client, but the repository still loads it internally regardless).
- **`WorkflowRepositoryInterface::summaries()`** defaults to `limit = 50` and is indexed on `workflow_type` — reasonable for a moderate number of instances, but no test has verified behaviour or query plan at, say, tens of thousands of Company Registration instances.
- **`dbDelta()`-based migration** (`Migrator::migrate()`) re-evaluates the full schema on every activation. For two small, well-indexed tables this is inexpensive today, but this has not been measured against a large, already-populated `bizhub_workflow_transitions` table (e.g. what happens to activation time after a busy production site has accumulated years of transitions).

## What should be added, and how

Once a real WordPress/MySQL test environment exists (see `Acceptance-Tests.md`), add: a seeded-database benchmark (using a seeder per `docs/database/Seeders.md`, once one exists) measuring `find()`/`summaries()`/`history()` latency at realistic and stress-test row counts; a REST-layer load test (e.g. `k6` or `wrk` against the four endpoints) to characterise request throughput and latency under concurrent load; and a migration-time benchmark comparing `Migrator::migrate()`'s duration on an empty versus a heavily-populated schema, since this directly affects perceived downtime during a plugin update on a busy production site.
